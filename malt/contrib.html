MALT

    I currently provide a first version of MALT, and, of course, I have some ideas on what to improve but don't have the time to do it myself now. If you are interested to contribute you can find here some starting points which might motivate you. I will be pleased to provide help, answer questions, detailed ideas, code reviews and to merge your great stuff. Be also welcome to provide your own ideas.

    You can look on the GitHub repository.

    You can also join the google group to discuss ideas and feedback

    In the GUI

    Angular cleanup

    I'm not an Angular expert so there is room for improvement on this topic in the web code.

    Export on all charts

    Currently not all charts support export functions.

    Better look and feel for main page

    The main page of MALT provides global statistics on your execution. It might be cool to get a list view with some logos for each metric instead of the current flat table.

    Improve help

    Ideally, I wanted to get documentation of all the features provided by the interface linked to some simple example of bad pattern it helps to detect.

    KCachgrind improvement

    Will be interesting to patch kcachgrind to use human readable units (KB, GB...) and also to support metrics with min/max inclusive functions instead of the sum. So we can use all MALT metrics in it.

    Reduce on class/libraries

    In source view the allocation metrics are reduced on functions, but, it can also (as kcachgrind to it well) reduce them on binary objects and classes.

    Translantion

    Could be really cool to get translations for the GUI and command-line tool.

    Trace file

    The trace file feature is not aimed to be enabled by default due to overhead and IO issues on big parallel applications. But it can help to go deeper in analysis when you find interesting spots. In the GUI it might appear as "deeper analysis" buttons on some elements to extract some content on demand from the trace file if available.

    Add view in the GUI

    The backend can generate some trace file but the data is not shown in the current GUI.

    Fragmentation

    The command line trace analysis tool can export some information about fragmentation by comparing two time-points. Might be interesting to get access to it in the GUI.

    What is allocated on timestemp

    In the GUI, on time charts, I wanted to be able to click on the chart to get the list of allocations which fill the memory at this point. The data are mostly already available.

    Time charts on call site

    Thanks to trace we can provide detailed time charts when clicking on a specific call site.

    Metric distribution on call site

    Thanks to trace we can provide a more detailed distribution chart of some metrics to view properties of call site allocations.

    Scatter plot f(size,lifetime)

    This is interesting to observe for some applications. We can extract some bad pattern quickly with this. Can only be done with trace as we might need to build a plot with millions of points.

    Backend

    Move out valgrind compatibility

    It might be cleaner to generate the callgrind files from the json one instead of doing it into the backend itself. It removes some unneeded constrains on internal stack representation.

    NUMA, physical/virtual

    Will be interesting to play with move_page() function on Linux to get NUMA and physical/virtual ratio on large segments at free time.

    use of libdwarf

    Currently MALT use addr2line to match the call site address to source code. It might be better for performance to use directly libdwarf.

    Multi-process summary

    In case you run a multi-process application like MPI, might be interesting to build a summary file with global statistics and export it in the webview with links to jump from one process to another.

    Hints about page fault costs

    When you allocate a large segment most of the cost comes from page faults to fill it. Provide some estimation of the cost.

    mmap profiles

    Currently MALT ignores mmap, there is the wrapping function but it requires to distinguish indirect call by malloc from user one and MALT one.

    Memory accesses

    Splitting the execution time in time slots we can use mprotect() to give an idea on dynamic memory size in use (accessed) over time.

    Unused before free

    Track time of last access before free, notify if too long compared to lifetime.

    Use backend V2

    What is V2

    I already completely rewrite a new version (see branch v2 in git) of the backend to make the code clean. It produces smaller and cleaner json file. Can also be extended to other metrics more easily.

    Still segfaults

    In current state I still get a segfault maybe due to a bug in the internal allocator if someone wants to search for it.

    Need to port the GUI

    The stack modeling is changed with this version and requires to port the GUI to take this in account. It might make the thing easier in the GUI code but require some work.

    Documentation

    The backend V2 is more abstract and can be reused for other kind of analysis. It will be good to have a proper documentation on how to proceed.

    Offload to thread

    Could be interesting to try to offload the status updating to another thread to not slow down the current one.

    Integration with other tools

    Valgrind

    In some cases, the backtrace approach is really slow, so it could be interesting to be integrated also as a valgrind tool.

    Pintool / Maqao

    There are already some preliminary examples to integrate binary instrumentation tools but it needs some work to become usable.

    Curses GUI

    Could be interesting to also get a curses GUI in NodeJS.
